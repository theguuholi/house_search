# ADR-007: Versioned Deterministic Ranking Formula

## Status

Accepted

## Date

2026-07-15

## Context

The PRD requires stable ranking, exact-before-near ordering, unknown values that
never count as matches, and explanations that can be audited. Ranking weights
must not change historical results silently.

## Decision

Eligibility first evaluates every broker-confirmed hard requirement. Only
eligible exact matches enter the exact ranking. Near-matches retain the complete
set of failed hard requirements and are ranked separately.

Ranking version `v1` uses a fixed 100-point formula: confirmed preference
matches contribute 55 points, freshness 15, evidence completeness 15, and
source confidence 15. Preference points are distributed across confirmed
preferences with explicit per-criterion contributions; unknown values receive
zero. Stable ties resolve by newest verification, highest completeness, highest
source confidence, then property-cluster UUID. Persist the ranking version,
score, and criterion-level breakdown with each match snapshot.

## Alternatives Considered

### Alternative 1: Original category weights

- **Description**: Allocate fixed points to price, location, features, and quality.
- **Pros**: Familiar real-estate categories.
- **Cons**: Can reward a hard criterion twice and underrepresent broker-classified preferences.
- **Why rejected**: Eligibility and ranking must follow the broker's classification.

### Alternative 2: Administrator-configurable weights

- **Description**: Allow live editing of ranking weights.
- **Pros**: Fast pilot experimentation.
- **Cons**: Adds validation, permissions, replay, and historical-comparison complexity.
- **Why rejected**: Code-versioned changes with regression fixtures are safer for the MVP.

## Consequences

### Positive

- Identical inputs and versions always produce identical order.
- Scores remain explainable at criterion level.
- Historical recommendations preserve their original behavior.

### Negative

- Weight experiments require a code and ranking-version change.
- A 55-point preference pool can compress differences when few preferences exist.

### Risks

- Weight selection may not predict usefulness; pilot feedback must be reviewed against versioned fixtures before `v2`.
- UUID tie-breaking is arbitrary but stable; the UI must not describe it as relevance.

## Implementation Notes

Implement ranking as pure functions over immutable structs. Add property-based
tests for bounds, determinism, unknown handling, and exact-before-near invariants.
