# ADR-006: Provider-Neutral Sagents Boundary

## Status

Accepted

## Date

2026-07-15

## Context

Natural-language intake and optional recommendation narratives require an LLM,
but criteria, eligibility, ranking, billing, and evidence integrity must remain
deterministic. No production provider or model has been selected, and tests
must not call external AI services.

## Decision

Use one constrained Sagents conversational agent behind a HouseSearch-owned
`HouseSearch.AI.Provider` behaviour. Select the provider and model through
runtime configuration. Tests use a deterministic fake provider.

The agent may propose structured criteria, request one clarification, and
generate narrative from a bounded verified evidence payload. Application code
validates every structured response. The agent receives no HTTP, SQL, billing,
source-management, or arbitrary job tools. Deterministic factual reasons remain
available when Sagents or the configured provider fails.

## Alternatives Considered

### Alternative 1: Couple directly to one provider and model

- **Description**: Use provider-specific request and response types throughout the application.
- **Pros**: Less adapter code initially.
- **Cons**: Couples domain tests and migrations to a volatile external contract.
- **Why rejected**: Provider choice is not settled and is not a domain concern.

### Alternative 2: Omit AI from the MVP

- **Description**: Use forms and deterministic templates only.
- **Pros**: Lowest integration risk.
- **Cons**: Does not deliver the conversational intake promised by the PRD.
- **Why rejected**: AI is valuable at the bounded language edges.

## Consequences

### Positive

- Provider and model can change without changing domain interfaces.
- Tests are deterministic and offline.
- AI outages degrade to factual UI rather than blocking research.

### Negative

- Provider capabilities must fit a common structured-output contract.
- Prompt, schema, and model versions require explicit persistence.

### Risks

- A provider may return unsupported facts; schema validation and evidence projection must reject them.
- Provider-neutral design can become overly abstract; support only the operations required by this PRD.

## Implementation Notes

Persist prompt version, schema version, provider key, model identifier, request
id, latency, token usage, and status for each call. Never persist buyer contact
data in prompts or traces.
