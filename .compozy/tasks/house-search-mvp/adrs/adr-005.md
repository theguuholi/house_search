# ADR-005: Generated Authentication with Account Memberships and Invitations

## Status

Accepted

## Date

2026-07-15

## Context

The repository contains Phoenix-generated email/password authentication,
including users, session tokens, confirmation, reset, and LiveView auth hooks.
The PRD prohibits public registration, requires administrator invitations and
suspension, and attributes cases and usage to broker accounts rather than only
to login identities.

## Decision

Retain the existing `phx.gen.auth` implementation as the authentication
foundation. Remove public registration from the router and expose user creation
only through an administrator-issued, single-use, expiring invitation. An
accepted invitation creates or activates the user, confirms the invited email,
and establishes a password through the generated credential rules.

Add `accounts` and `memberships`. Cases, subscriptions, and ledger entries
belong to accounts. Memberships authorize account access. A system-level role
on the user grants administrator capabilities; membership role grants broker
account access. Suspension revokes active sessions and is rechecked before
every billable or mutating action.

## Alternatives Considered

### Alternative 1: Direct user ownership

- **Description**: Attach every case and usage event directly to a user.
- **Pros**: Fewer tables.
- **Cons**: Conflates authentication with billing ownership and complicates cross-account administration.
- **Why rejected**: Account-scoped audit and future team membership justify the boundary now.

### Alternative 2: Owner-only account

- **Description**: Give each account one immutable owner without memberships.
- **Pros**: Preserves an account billing boundary with less authorization logic.
- **Cons**: Requires a later ownership migration to support assistants or agencies.
- **Why rejected**: A membership table is small and prevents destructive evolution.

## Consequences

### Positive

- Existing generated security behavior and tests remain useful.
- Authentication, authorization, and billing ownership have distinct boundaries.
- Invitation replay, suspension, and session invalidation are auditable.

### Negative

- Generated public-registration code must be adapted rather than used unchanged.
- Every account operation requires membership authorization.

### Risks

- Conflicting user and membership roles could create privilege ambiguity; system admin and account access must remain separate.
- Suspension checks can be missed; all mutating context entry points must enforce them centrally.

## Implementation Notes

Store hashed invitation tokens, invited email, inviter, status, and expiry.
Enforce one pending invitation per normalized email. Use optimistic locking for
administrator edits and disconnect LiveView sessions when a user is suspended.
