# ADR-004: Phoenix Modular Monolith with Durable Oban Orchestration

## Status

Accepted

## Date

2026-07-15

## Context

HouseSearch must expose progressive LiveView results, survive browser and
application restarts, isolate source failures, enforce a ten-minute research
deadline, and make retries idempotent. The repository is a Phoenix scaffold
with PostgreSQL and PubSub but no implemented domain contexts or job runtime.

## Decision

Implement the MVP as one Phoenix modular monolith backed by PostgreSQL. Divide
the domain into `Accounts`, `Sources`, `Ingestion`, `Listings`, `SearchCases`,
`Ranking`, `AI`, and `Usage` contexts. Contexts communicate through public APIs
and never query another context's tables directly.

Use Oban OSS for durable ingestion, recomputation, deadline finalization,
credits, and maintenance jobs. Jobs persist business progress before
broadcasting a PubSub invalidation. LiveViews always reload persisted state;
PubSub messages never contain the only copy of a result.

## Alternatives Considered

### Alternative 1: In-process OTP tasks

- **Description**: Run refresh and ranking work with `Task`, supervisors, and timers.
- **Pros**: No job dependency and minimal initial setup.
- **Cons**: Work and deadlines are harder to recover after deploys or crashes.
- **Why rejected**: Durable retries and persisted orchestration are core MVP behavior.

### Alternative 2: Separate ingestion service

- **Description**: Deploy source collection independently from Phoenix.
- **Pros**: Independent scaling and fault isolation.
- **Cons**: Adds deployment, contracts, tracing, and consistency overhead for a five-broker pilot.
- **Why rejected**: The pilot does not justify a distributed system.

## Consequences

### Positive

- One deployment retains explicit domain boundaries.
- Jobs, deadlines, and retries survive application restarts.
- Persisted state makes LiveView reconnection deterministic.

### Negative

- Collection load shares the application database and release.
- Oban queues and database retention require operations work.

### Risks

- Context boundaries may erode; tests and code review must reject cross-context table access.
- Large source batches may contend with UI queries; bounded batches and queue limits mitigate this.

## Implementation Notes

Start Oban under `HouseSearch.Application`. Configure distinct queues for
on-demand ingestion, scheduled ingestion, normalization, recommendations, and
maintenance. Use unique job arguments and database constraints for business
idempotency; Oban uniqueness is only an enqueue optimization.
